# simplees 完整 0 到 1 开发步骤记录

本文记录当前项目从空骨架到“SQLite + SQLAlchemy + Elasticsearch 商品搜索基础链路”跑通的完整步骤。这里默认项目一开始就选择 SQLAlchemy，不记录中途技术选型调整过程。

当前目标不是一次性完成全部接口，而是先把基础链路打通：

```text
配置文件
  ↓
SQLAlchemy 连接 SQLite
  ↓
创建商品/品牌/类目表
  ↓
初始化 100 条商品测试数据
  ↓
连接 Elasticsearch
  ↓
创建 ES 商品搜索索引
  ↓
同步 SQLite 商品数据到 ES
  ↓
测试 ES 搜索
  ↓
封装搜索服务逻辑
```

## 1. 创建依赖文件 requirements.txt

第一步是在项目根目录创建：

```text
requirements.txt
```

内容如下：

```txt
fastapi==0.136.3
uvicorn[standard]==0.48.0
elasticsearch>=8.0.0,<9.0.0
pydantic==2.13.4
python-dotenv==1.2.2
SQLAlchemy>=2.0.0,<3.0.0
```

每个依赖的作用：

| 依赖 | 作用 |
|---|---|
| `fastapi` | 后续提供 HTTP 接口 |
| `uvicorn` | 启动 FastAPI 服务 |
| `elasticsearch` | Python 连接 ES |
| `pydantic` | 定义接口返回结构 |
| `python-dotenv` | 读取 `.env` 配置文件 |
| `SQLAlchemy` | 通过 ORM 操作 SQLite 数据库 |

安装依赖：

```bash
pip install -r requirements.txt
```

这一步的意义：先把项目要用的工具箱固定下来，后续所有代码都围绕这些库展开。

## 3. 创建本地配置文件 .env

在项目根目录创建：

```text
.env
```

内容如下：

```env
# 应用名称，只用于日志、启动信息等展示
APP_NAME=simplees

# 接口服务监听地址
# 127.0.0.1 表示只允许本机访问，适合本地开发
APP_HOST=127.0.0.1

# 接口服务端口
# 启动后访问地址会是 http://127.0.0.1:8000
APP_PORT=8000

# 数据库连接地址
# 当前使用 SQLite：sqlite:///./data/simplees.db
# 以后如果换 MySQL / PostgreSQL，只需要改这个连接地址和安装对应驱动
DATABASE_URL=sqlite:///./data/simplees.db

# Elasticsearch 服务地址
# 如果你本地用 Docker 启动 ES，通常就是 http://127.0.0.1:9200
ES_HOST=http://127.0.0.1:9200

# Elasticsearch 用户名
# 本地开发如果没有开启账号密码，可以留空
ES_USERNAME=

# Elasticsearch 密码
# 本地开发如果没有开启账号密码，可以留空
ES_PASSWORD=

# 商品搜索使用的 ES 索引名
# 后面商品、品牌、类目会合并成商品搜索文档写入这个索引
ES_PRODUCT_INDEX=product_search

# 请求 ES 的超时时间，单位：秒
# 超过这个时间 ES 还没响应，就认为请求失败
ES_REQUEST_TIMEOUT_SECONDS=5

# 搜索接口默认每页返回多少条
SEARCH_DEFAULT_PAGE_SIZE=10

# 搜索接口最多允许每页返回多少条
# 防止一次请求返回太多数据
SEARCH_MAX_PAGE_SIZE=50
```

这一步的意义：把数据库地址、ES 地址、索引名、分页大小这些配置从代码里拿出来，后续修改配置时不需要改 Python 代码。

## 4. 创建 app/config.py

在 `app/` 目录下创建：

```text
app/config.py
```

这个文件负责读取 `.env`，并整理成 `settings` 对象。

核心作用：

```text
.env 配置
  ↓
app/config.py 读取
  ↓

```

后续其他文件使用方式：

```python

```

这一步的意义：统一配置入口，避免每个文件都自己读取环境变量。

## 5. 创建 app/database.py

在 `app/` 目录下创建：

```text
app/database.py
```

这个文件负责创建 SQLAlchemy 数据库入口，主要包含：

| 对象 | 作用 |
|---|---|
| `engine` | SQLAlchemy 数据库引擎，负责底层数据库连接 |
| `SessionLocal` | 数据库会话工厂，每次操作数据库时创建 Session |
| `Base` | 所有 ORM 模型类的父类 |
| `get_db()` | 后续给 FastAPI 接口使用的数据库依赖函数 |

当前使用的是同步 SQLAlchemy，不使用异步。原因是本项目接口数量少、数据量小、学习阶段优先保证链路清晰。

这一步的意义：建立数据库访问的统一入口，为后续模型、建表、查询做准备。

## 6. 创建 app/models.py

在 `app/` 目录下创建：

```text
app/models.py
```

这个文件用 SQLAlchemy ORM 模型描述数据库表。

当前有三张表：

| Python 类 | 数据库表 | 说明 |
|---|---|---|
| `Brand` | `brands` | 品牌表 |
| `Category` | `categories` | 类目表 |
| `Product` | `products` | 商品表 |

主键统一使用 UUID 字符串，不使用自增 ID。

关系设计：

```text
Brand 1 - N Product
Category 1 - N Product
```

也就是：

```text
一个品牌下面有多个商品
一个类目下面有多个商品
一个商品只属于一个品牌和一个类目
```

这一步的意义：用 Python 类描述数据库结构，后面可以通过 ORM 操作数据，而不是手写大量 SQL。

## 10. 创建 app/es_index.py

在 `app/` 目录下创建：

```text
app/es_index.py
```

这个文件负责用代码创建 ES 商品搜索索引：

```text
product_search
```

ES 索引字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `product_id` | `keyword` | 商品 UUID，用于回查 SQLite |
| `product_name` | `text` + `keyword` | 商品名称 |
| `brand_id` | `keyword` | 品牌 UUID |
| `brand_name` | `text` + `keyword` | 品牌名称 |
| `category_id` | `keyword` | 类目 UUID |
| `category_name` | `text` + `keyword` | 类目名称 |
| `search_text` | `text` | 商品名 + 品牌名 + 类目名组合字段 |
| `status` | `keyword` | 商品状态 |
| `updated_at` | `date` | 更新时间 |

执行命令：

```bash
python3 -m app.es_index
```

可能输出：

```text
ES 索引创建完成：product_search
```

或：

```text
ES 索引已存在：product_search
```

这一步的意义：不要在 ES 控制面板手动点配置，而是通过代码创建索引。这样项目可复现，换电脑或重建环境时可以直接执行脚本恢复索引结构。

## 11. 创建 app/es_sync.py

在 `app/` 目录下创建：

```text
app/es_sync.py
```

这个文件负责把 SQLite 中的商品数据同步到 ES。

同步规则：

```text
SQLite 是主数据源
ES 是搜索副本
```

每个商品对应 ES 中一条 document：

```text
products.id = ES document _id
```

商品同步到 ES 的文档结构：

```json
{
  "product_id": "商品 UUID",
  "product_name": "商品名称",
  "brand_id": "品牌 UUID",
  "brand_name": "品牌名称",
  "category_id": "类目 UUID",
  "category_name": "类目名称",
  "search_text": "商品名称 品牌名称 类目名称",
  "status": "active",
  "updated_at": "更新时间"
}
```

同步行为：

| 商品状态 | ES 行为 |
|---|---|
| 非 `deleted` | 写入或覆盖 ES document |
| `deleted` | 删除 ES document |

执行命令：

```bash
python3 -m app.es_sync
```

成功后会看到：

```text
商品同步完成，共同步 100 条商品到 ES
```

这一步的意义：把数据库里的真实商品数据转换成 ES 可以搜索的索引数据。

## 13. 创建 app/services.py

在 `app/` 目录下创建：

```text
app/services.py
```

这个文件负责封装业务逻辑，不让接口层直接写 ES 查询和数据库查询细节。

当前包含几个核心函数：

| 函数 | 作用 |
|---|---|
| `normalize_page()` | 修正页码，最小为 1 |
| `normalize_size()` | 修正每页数量，不能超过配置上限 |
| `search_product_ids_from_es()` | 从 ES 搜索商品 UUID |
| `get_products_by_ids()` | 根据 UUID 回查 SQLite，并保持 ES 排序 |
| `search_products()` | 商品搜索总入口：先查 ES，再查 SQLite |
| `get_product_by_id()` | 根据 UUID 直接查 SQLite 商品详情 |

搜索流程：

```text
用户输入关键词
  ↓
ES multi_match 搜索商品名、品牌名、类目名、组合字段
  ↓
ES 返回商品 UUID 列表和 total
  ↓
SQLite 根据 UUID 查询商品详情
  ↓
按 ES 相关性顺序返回商品列表
```

这里使用的 ES Query DSL 可以类比 SQL：

```sql
SELECT product_id
FROM product_search
WHERE status = 'active'
  AND (
    product_name LIKE '%关键词%'
    OR brand_name LIKE '%关键词%'
    OR category_name LIKE '%关键词%'
    OR search_text LIKE '%关键词%'
  )
LIMIT size OFFSET from;
```

对应 ES 结构是：

```text
bool
  must
    multi_match
  filter
    term status=active
```

这一步的意义：把“怎么搜索商品”沉淀成服务层方法，后面 FastAPI 接口只需要调用服务层，不需要关心 ES DSL 细节。
