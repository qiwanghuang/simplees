# simplees 完整 0 到 1 开发步骤记录

本文记录当前项目从空骨架到“SQLite + SQLAlchemy + Elasticsearch 商品搜索基础链路”跑通的完整步骤。这里默认项目一开始就选择 SQLAlchemy，不记录中途技术选型调整过程。

当前目标不是一次性完成全部接口，而是先把基础链路打通：

```text
配置文件
  ↓
SQLAlchemy 连接 SQLite
  ↓
创建商品/品牌/类目表
  ↓
初始化 100 条商品测试数据
  ↓
连接 Elasticsearch
  ↓
创建 ES 商品搜索索引
  ↓
同步 SQLite 商品数据到 ES
  ↓
测试 ES 搜索
  ↓
封装搜索服务逻辑
```

## 3. 创建本地配置文件 .env

在项目根目录创建：

```text

```

内容如下：

```env
# 应用名称，只用于日志、启动信息等展示
APP_NAME=simplees

# 接口服务监听地址
# 127.0.0.1 表示只允许本机访问，适合本地开发

# 数据库连接地址

# 如果你本地用 Docker 启动 ES，通常就是 http://127.0.0.1:9200
ES_HOST=http://127.0.0.1:9200

# Elasticsearch 用户名
# 本地开发如果没有开启账号密码，可以留空
ES_USERNAME=

# Elasticsearch 密码
# 本地开发如果没有开启账号密码，可以留空
ES_PASSWORD=

# 商品搜索使用的 ES 索引名
# 后面商品、品牌、类目会合并成商品搜索文档写入这个索引
```

这一步的意义：把数据库地址、ES 地址、索引名、分页大小这些配置从代码里拿出来，后续修改配置时不需要改 Python 代码。

## 11. 创建 app/es_sync.py

在 `app/` 目录下创建：

```text
app/es_sync.py
```

这个文件负责把 SQLite 中的商品数据同步到 ES。

同步规则：

```text
SQLite 是主数据源
ES 是搜索副本
```

每个商品对应 ES 中一条 document：

```text
products.id = ES document _id
```

商品同步到 ES 的文档结构：

```json
{
  "product_id": "商品 UUID",
  "product_name": "商品名称",
  "brand_id": "品牌 UUID",
  "brand_name": "品牌名称",
  "category_id": "类目 UUID",
  "category_name": "类目名称",
  "search_text": "商品名称 品牌名称 类目名称",
  "status": "active",
  "updated_at": "更新时间"
}
```

同步行为：

| 商品状态 | ES 行为 |
|---|---|
| 非 `deleted` | 写入或覆盖 ES document |
| `deleted` | 删除 ES document |

执行命令：

```bash
python3 -m app.es_sync
```

成功后会看到：

```text
商品同步完成，共同步 100 条商品到 ES
```

这一步的意义：把数据库里的真实商品数据转换成 ES 可以搜索的索引数据。

## 13. 创建 app/services.py

在 `app/` 目录下创建：

```text
app/services.py
```

这个文件负责封装业务逻辑，不让接口层直接写 ES 查询和数据库查询细节。

当前包含几个核心函数：

| 函数 | 作用 |
|---|---|
| `normalize_page()` | 修正页码，最小为 1 |
| `normalize_size()` | 修正每页数量，不能超过配置上限 |
| `normalize_keyword()` | 清理搜索关键词前后空格 |
| `compact_keyword()` | 去掉关键词空白，用于商品名完整包含匹配 |
| `search_product_ids_from_es()` | 从 ES 搜索商品 UUID |
| `get_products_by_ids()` | 根据 UUID 回查 SQLite，并保持 ES 排序 |
| `search_products()` | 商品搜索总入口：先查 ES，再查 SQLite |
| `list_products()` | 商品分页列表：直接查 SQLite，不走 ES |
| `get_product_by_id()` | 根据 UUID 直接查 SQLite 商品详情 |

搜索流程：

```text
用户输入关键词
  ↓
ES 搜索商品名和组合字段
  ↓
ES 返回商品 UUID 列表和 total
  ↓
SQLite 根据 UUID 查询商品详情
  ↓
按 ES 相关性顺序返回商品列表
```

这里使用的 ES Query DSL 可以类比 SQL：

```sql
SELECT product_id
FROM product_search
WHERE status = 'active'
  AND (
    product_name LIKE '%完整关键词%'
    OR (product_name/search_text 同时命中分词后的全部关键词)
  )
LIMIT size OFFSET from;
```

对应 ES 结构是：

```text
bool
  should
    wildcard product_name.keyword
    multi_match operator=and
  minimum_should_match=1
  filter
    term status=active
```

这里后来做过一次搜索效果优化。原来的 `multi_match` 默认偏 OR 逻辑，搜索“小米手机”时会出现“小米手表”和“vivo手机”这种只命中部分词的结果。现在改成两层匹配：

1. `product_name.keyword` 完整包含匹配，优先返回商品名连续包含完整关键词的商品。
2. `multi_match` 使用 `operator=and`，要求分词后的关键词都命中。

当前验证结果：

```text
小米手机 -> total=0，因为测试数据里没有小米手机，不再混入小米手表或 vivo 手机
vivo手机 -> 返回 vivo 手机
小米 -> 返回小米品牌商品
手机 -> 返回手机类目商品
```

这一步的意义：把“怎么搜索商品”沉淀成服务层方法，后面 FastAPI 接口只需要调用服务层，不需要关心 ES DSL 细节。

## 20. ES 索引状态说明

在 ES 控制面板里，`product_search` 可能显示：

```text
health = yellow
status = open
nodes = 1
data nodes = 1
active primary shards = 1
active shards = 1
unassigned shards = 1
```

这些参数含义：

| 参数 | 说明 |
|---|---|
| `health=yellow` | 主分片可用，副本分片未分配 |
| `status=open` | 索引已打开，可以读写 |
| `nodes=1` | 当前 ES 集群只有 1 个节点 |
| `data nodes=1` | 有 1 个数据节点 |
| `active primary shards=1` | 有 1 个主分片正常工作 |
| `active shards=1` | 当前可用分片总数是 1 |
| `unassigned shards=1` | 有 1 个副本分片没有被分配 |

本地单节点 ES 出现 `yellow` 很常见。原因是索引通常默认有 1 个副本分片，但副本不能和主分片放在同一个节点上，本地只有一个节点，所以副本无法分配。

判断标准：

```text
green  = 主分片和副本分片都正常
yellow = 主分片正常，副本分片缺失
red    = 主分片异常，数据可能不可用
```

当前 `yellow` 不影响本地开发、写入和搜索。
